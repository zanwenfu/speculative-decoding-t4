\begin{abstract}
Autoregressive decoding remains a primary latency bottleneck in large language model (LLM) inference due to its inherent sequential dependency and limited token-level parallelism.
Speculative decoding mitigates this constraint by using lightweight mechanisms to propose multiple candidate tokens that a larger target model verifies in fewer forward passes, preserving exact decoding semantics~\cite{leviathan2023speculative}.
Although the foundational algorithm admits substantial theoretical acceleration, practical implementations can exhibit inconsistent gains due to KV-cache management, GPU memory behavior, and framework-level scheduling constraints~\cite{vllm_paper,vllm_docs,trtllm_docs}.

This paper presents a systems-oriented comparative study of three speculative decoding families: linear draft-verify~\cite{leviathan2023speculative}, tree-structured speculation~\cite{miao2024specinfer,chen2024sequoia}, and retrieval-based model-free methods~\cite{he2024rest,saxena2023pld}.
I develop a unified theoretical framework that extends the standard acceptance-based speedup model to each family, exposing fundamentally different overhead profiles: draft model inference cost, tree-structured KV-cache complexity, and CPU-side retrieval latency.
I then conduct controlled experiments on a T4 GPU to measure end-to-end performance and attribute the gap between theoretical predictions and measured speedups to concrete system-level bottlenecks.
Preliminary results from linear draft-verify and prompt lookup decoding are reported, with tree-structured experiments in progress.
\end{abstract}

\section{Introduction}

Transformer-based LLMs generate text autoregressively, producing one token at a time conditioned on previous outputs.
Even with KV caching and optimized attention implementations such as FlashAttention~\cite{flashattention}, decoding remains sequential at the token level, and the per-token loop is frequently dominated by memory movement rather than compute at small batch sizes.

Speculative decoding reduces the number of expensive target-model forward passes by having a faster mechanism propose multiple candidate tokens that the target model verifies in parallel~\cite{leviathan2023speculative}.
Under idealized assumptions, expected speedup increases with proposal length and acceptance rate.
However, these assumptions omit system effects---draft inference cost, verification overhead, KV-cache rewind for rejected tokens, and scheduling latency---that can dominate end-to-end performance.

\textbf{Observation.}
Speculative decoding is not one technique but a \emph{family} of methods that make fundamentally different bets about where to spend system resources.
Linear draft-verify methods~\cite{leviathan2023speculative} trade GPU compute for a small draft model.
Tree-structured methods~\cite{miao2024specinfer,chen2024sequoia} expand the speculation space but increase KV-cache footprint and verification complexity.
Retrieval-based methods~\cite{he2024rest,saxena2023pld} eliminate the draft model entirely, replacing GPU-side inference with CPU-side n-gram lookup.
Each family exhibits a distinct overhead profile, yet most prior work evaluates methods in isolation, making cross-family comparison difficult.

This study investigates speculative decoding as a systems-performance phenomenon through three central questions:
(1)~How closely do empirical speedups match acceptance-based theoretical predictions across families?
(2)~Which system-level bottleneck---draft compute, KV-cache management, or retrieval latency---dominates in each family?
(3)~Under what deployment regimes (hardware, model size, decoding temperature, prompt domain) does each family offer a practical advantage?

\section{Background and Taxonomy}

I organize speculative decoding methods into three families based on their draft generation mechanism and corresponding system overhead profile.

\subsection{Family 1: Linear Draft-Verify}

Leviathan et al.~\cite{leviathan2023speculative} formalized speculative decoding as an exact acceleration technique.
A draft model $M_d$ autoregressively proposes $k$ tokens, and the target model $M_t$ verifies them in a single forward pass.
The algorithm commits the longest accepted prefix and repeats.
Theoretical analysis relates expected speedup to proposal length~$k$, acceptance rate~$\alpha$, and relative model costs.

The system overhead is concentrated in draft model inference: $M_d$ must run $k$ sequential forward passes, each requiring GPU compute and KV-cache updates.
On rejection, the target model's KV-cache must be rewound to discard entries for rejected tokens---a bookkeeping operation whose cost depends on the cache implementation (contiguous vs.\ paged storage).

HuggingFace Transformers implements this paradigm via assisted generation~\cite{hf_assisted}, supporting both fixed and adaptive speculation lengths.
The \texttt{num\_assistant\_tokens} parameter controls~$k$, and the \texttt{num\_assistant\_tokens\_schedule} parameter enables heuristic adjustment based on acceptance history.

\subsection{Family 2: Tree-Structured Speculation}

Rather than proposing a single chain of $k$ tokens, tree-structured methods explore multiple candidate paths simultaneously.

\textbf{SpecInfer}~\cite{miao2024specinfer} uses one or more small speculative models to generate candidate tokens organized as a token tree.
The target model verifies all candidates in parallel using a tree-based attention mask, enabling a single forward pass to validate an entire tree of speculation.

\textbf{Sequoia}~\cite{chen2024sequoia} improves upon SpecInfer in three ways.
First, it introduces a dynamic programming algorithm to find the optimal tree topology for a given speculation budget, maximizing expected accepted tokens.
Second, it uses a sampling-without-replacement verification procedure that is robust across decoding temperatures---unlike SpecInfer and top-$k$ sampling, which can repeatedly sample incorrect tokens at low temperatures.
Third, it introduces a hardware-aware tree optimizer that selects tree size and depth based on the target hardware's compute-to-bandwidth ratio.

Single-model variants such as Medusa~\cite{cai2024medusa} and Hydra~\cite{ankner2024hydra} augment the target model with extra prediction heads instead of a separate draft, eliminating cross-model overhead at the cost of model-specific training; I do not evaluate these in this study.

The key system difference from linear methods is KV-cache complexity.
Tree verification requires storing KV entries for \emph{all} nodes in the speculation tree, not just a linear chain.
Verification cost grows with tree size, and rejected branches require selective cache pruning rather than simple truncation.
Sequoia reports optimal tree sizes of ${\sim}64$ nodes for on-chip inference and 768+ for offloading~\cite{chen2024sequoia}.

\subsection{Family 3: Retrieval-Based / Model-Free Methods}
\label{sec:family3}

Retrieval-based methods eliminate the draft model entirely, instead proposing candidate tokens by matching patterns in existing text.

\textbf{Prompt Lookup Decoding (PLD)}~\cite{saxena2023pld} finds overlapping n-grams between the partially generated output and the input prompt, using matched continuations as candidate tokens.
It requires no additional model, no training, and zero extra GPU memory.
HuggingFace implements this as a single parameter: \texttt{prompt\_lookup\_num\_tokens}.

\textbf{REST}~\cite{he2024rest} generalizes this to retrieval from an external datastore using suffix arrays.
Given the current generation context, REST performs suffix matching against a large text corpus, constructs a trie of candidate continuations, prunes low-frequency branches, and feeds the remaining tree to the target model for verification via tree attention.

\textbf{SAM-Decoding}~\cite{hu2024samdecoding} replaces the suffix array with a suffix automaton, achieving $O(1)$ amortized time for longest-suffix matching and integrating with model-based drafters like EAGLE for hybrid speculation.

The overhead profile is fundamentally different: draft cost~$C_d$ is replaced by retrieval cost~$C_r$, which is CPU-bound (suffix matching, trie construction) rather than GPU-bound.
This means retrieval methods add essentially zero GPU memory overhead but are highly domain-dependent: acceptance rates are high when the output overlaps with the datastore (e.g., summarization, RAG) but low for open-ended generation.

\section{Extended Theoretical Framework}
\label{sec:theory}

\subsection{Linear Draft-Verify (Review)}

Following~\cite{leviathan2023speculative}, let $C_t$ denote the cost of one target forward pass, $C_d$ the cost per draft token, $k$ the proposal length, and $\alpha$ the empirical acceptance rate.
Each speculative iteration drafts $k$ tokens and performs one verification pass:
\[
C_{\text{iter}} = kC_d + C_t.
\]
The expected committed tokens per iteration is approximately $\alpha k$, yielding speedup over greedy decoding:
\begin{equation}
\label{eq:linear_speedup}
S_{\text{linear}} = \frac{\alpha k \cdot C_t}{kC_d + C_t}.
\end{equation}
This model predicts that speedup improves as $\alpha$ increases, $C_d/C_t$ decreases (smaller draft relative to target), and $k$ grows (more tokens proposed).
However, it omits a system overhead term $C_{\text{sys}}$ (cache rewind, synchronization, scheduling) that reduces realized speedup:
\begin{equation}
\label{eq:linear_speedup_real}
S_{\text{linear}}^{\text{real}} = \frac{\alpha k \cdot C_t}{kC_d + C_t + C_{\text{sys}}}.
\end{equation}
Quantifying $C_{\text{sys}}$ empirically is a primary goal of my experiments.

\subsection{Tree-Structured Speculation}

For tree-structured methods, the cost model changes in two ways.
First, drafting produces a tree of $n$ nodes (not a chain), with total cost $C_d^{\text{tree}}(n)$.
Second, verification cost grows with $n$ (tree attention over all nodes), unlike the approximately-constant cost in the linear case:
\[
C_{\text{iter}}^{\text{tree}} = C_d^{\text{tree}}(n) + C_t^{\text{tree}}(n).
\]
Sequoia's DP algorithm optimizes tree topology to maximize expected accepted tokens $\mathbb{E}[\tau(n)]$ for a given budget.
Speedup becomes:
\begin{equation}
\label{eq:tree_speedup}
S_{\text{tree}} = \frac{\mathbb{E}[\tau(n)] \cdot C_t}{C_d^{\text{tree}}(n) + C_t^{\text{tree}}(n) + C_{\text{sys}}^{\text{tree}}}.
\end{equation}
The system overhead $C_{\text{sys}}^{\text{tree}}$ is expected to exceed the linear case because rejected branches require selective KV-cache pruning rather than simple truncation.
Critically, $C_t^{\text{tree}}(n)$ grows with $n$, creating diminishing returns beyond a hardware-dependent optimal tree size~\cite{chen2024sequoia}.

\subsection{Retrieval-Based Methods}

For retrieval-based methods, draft cost is replaced by CPU-side retrieval cost:
$C_{\text{iter}}^{\text{ret}} = C_r + C_t^{\text{ret}}(m)$,
where $C_r$ is suffix matching and trie construction cost, and $m$ is the number of candidate tokens.
Since $C_r$ is CPU-bound, it adds zero GPU compute overhead---a fundamentally different resource tradeoff.
Speedup is:
\begin{equation}
\label{eq:ret_speedup}
S_{\text{ret}} = \frac{\mathbb{E}[\tau_r] \cdot C_t}{C_r + C_t^{\text{ret}}(m)}.
\end{equation}
The key variable is domain-dependent acceptance: $\mathbb{E}[\tau_r]$ is high for input-grounded tasks (summarization, RAG) but low for open-ended generation.
For prompt lookup decoding, $C_r$ is negligible (in-prompt string matching), making it essentially free when it succeeds and harmless when it fails.

\section{Experimental Design}

\subsection{Hardware and Environment}

All experiments run on a single NVIDIA T4 GPU (16\,GB VRAM, ${\sim}300$\,GB/s bandwidth) via Google Colab, with CUDA 12.8 and PyTorch 2.10.
The T4's limited bandwidth makes system-level effects (memory stalls, KV-cache pressure) more pronounced than on high-end hardware.

\subsection{Models}

I use the Llama 3.2 family for tokenizer consistency: Llama-3.2-3B (fp16, ${\sim}6$\,GB) as the target and Llama-3.2-1B (fp16, ${\sim}2$\,GB) as the draft model for Family~1.
To isolate the effect of quantization on the draft-to-target cost ratio, I additionally evaluate the 1B draft against Llama-3.1-8B in 4-bit quantization (parameter ratio 0.125 vs.\ 0.33).
Family~3 (prompt lookup) requires no draft model.
Family~2 (planned): Sequoia on T4 with model pair to be determined based on compatibility testing.

\subsection{Prompt Suite and Controlled Variables}

I use 10 prompts spanning five categories: general knowledge~(3), code generation~(2), summarization with high text repetition~(1), instruction following~(2), and long-context prefills~(2), enabling per-category acceptance rate analysis.
Generation is capped at 128 new tokens.

I fix precision (fp16), batch size (1, latency-focused), and framework (HuggingFace Transformers), and vary:
(i)~speculation length $k \in \{3, 5, 7, 10\}$;
(ii)~lookup window $n \in \{3, 5, 7, 10\}$;
(iii)~temperature $T \in \{0.0, 0.6, 0.8\}$.
For linear SD, I set \texttt{num\_assistant\_tokens\_schedule="constant"} to fix~$k$ and prevent heuristic confounds.

\subsection{Measurements}

For each configuration, I run 3 warmup iterations followed by 30 timed trials with proper CUDA synchronization (\texttt{torch.cuda.synchronize()}).
I report: mean and median throughput (tokens/sec), p95 tail latency, end-to-end wall-clock time, peak GPU memory, and speedup relative to the autoregressive baseline at the same temperature.
All timing uses \texttt{time.perf\_counter()} with explicit GPU synchronization to avoid measuring only kernel launch overhead.

\section{Preliminary Results}

All experiments use Llama-3.2-3B (fp16) as the target model on a T4 GPU (CUDA 12.8, PyTorch 2.10).
I report results from two configurations: (1)~Llama-3.2-1B as draft against the 3B target (parameter ratio~0.33), and (2)~the same 1B draft against Llama-3.1-8B in 4-bit quantization (parameter ratio~0.125) to isolate the effect of quantization on the effective cost ratio.

\subsection{Baseline Autoregressive Performance}

Under greedy decoding, the target model achieves 26.42 tokens/sec with a mean per-request latency of 4.275\,s for 128 generated tokens (standard deviation 1.48\,tok/s across prompts).
Peak GPU memory is 8{,}940\,MB.
This establishes the denominator for all speedup calculations.

When the draft model is insufficiently small, speculative decoding produces net slowdowns regardless of~$k$.

\subsection{Temperature Sensitivity}

Speculative decoding performance degrades under sampling: with the 1B/3B configuration at $k{=}5$, speedup drops from $0.98{\times}$ (greedy) to $0.77{\times}$ ($T{=}0.6$) and $0.83{\times}$ ($T{=}0.8$).
The 1B/8B-4bit configuration shows the same pattern more severely: $0.72{\times}$ at $T{=}0.6$.
Sampling reduces draft-target agreement, lowering acceptance rates and compounding the cost-ratio problem.
This motivates tree-structured methods such as Sequoia~\cite{chen2024sequoia}, which use sampling-without-replacement verification designed for temperature robustness.

\subsection{Theory vs.\ Practice: Quantitative Gap Analysis}

To close the loop between the theoretical framework (Section~\ref{sec:theory}) and measured results, I independently measure draft cost~$C_d$, target cost~$C_t$, and per-iteration acceptance rate~$\alpha$.

\textbf{Cost and acceptance measurement.}
I implement a manual speculative decoding loop to record per-iteration acceptance counts, and independently measure $C_d$ and $C_t$ via autoregressive generation timing.
The 3B target yields $C_t = 37.63$\,ms/token; the 1B draft yields $C_d = 22.42$\,ms/token, giving a measured cost ratio $C_d/C_t = 0.596$.
This is nearly double the parameter ratio~(0.333), revealing that on memory-bandwidth-bound hardware, per-token cost does not scale linearly with parameter count---kernel launch overhead and memory access patterns compress the gap.
